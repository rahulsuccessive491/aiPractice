# Database

## Driver state

- **Now (Sprint 1–3):** SQLite via `better-sqlite3`, file at `backend/data/portal.db`.
- **Target (Sprint 4 / prod):** PostgreSQL.
- All queries go through `backend/src/db.js`. **Never** import `better-sqlite3` directly in routes — the only legitimate caller is `db.js` and `migrate.js`. This keeps the Postgres swap to a single file.

## Schema (current)

Defined in `backend/src/migrations/001_initial_schema.sql`. Summary:

```
teams (id, name UNIQUE, description, created_at)
tags  (id, name UNIQUE, kind, created_at)            -- kind: domain | tech | tool | activity_type
users (id, email UNIQUE, password_hash, first_name, last_name,
       mobile, department, team_id → teams, role,
       tech_stack JSON[], ai_tools JSON[], bio,
       created_at, updated_at)
activities (id, user_id → users, activity_type, title, tool_used,
            domain, status, notes, activity_date,
            created_at, updated_at)
activity_tags (activity_id → activities, tag_id → tags, PK composite)
```

Indexes: `users(team_id)`, `users(department)`, `activities(user_id)`, `activities(activity_date)`.

## Sprint usage

- **Sprint 1:** `teams`, `tags`, `users`.
- **Sprint 2:** `activities`, `activity_tags`.
- **Sprint 3:** read-only aggregates over the above.

## Portability rules — keep migrations Postgres-safe

Stick to this checklist so the Sprint 4 migration is mechanical:

- Use **only** `TEXT`, `INTEGER`, `REAL`, `BLOB`. No SQLite-specific `DATETIME`/`BOOLEAN`.
- Booleans: store as `INTEGER` (`0` / `1`).
- Timestamps: store as ISO strings via `datetime('now')` for now; on Postgres these become `TIMESTAMPTZ DEFAULT NOW()`.
- Arrays / multi-value: store as JSON strings (`TEXT`). Use `db.parseJson()` on read. On Postgres, can move to `JSONB` later.
- Primary keys: `INTEGER PRIMARY KEY AUTOINCREMENT`. Pg equivalent: `BIGSERIAL PRIMARY KEY` or `GENERATED BY DEFAULT AS IDENTITY`.
- Foreign keys: always declared. SQLite needs `PRAGMA foreign_keys = ON` (set in `db.js`).
- `INSERT OR IGNORE` (SQLite) → `INSERT … ON CONFLICT DO NOTHING` (Postgres). Use seeded fixtures sparingly and keep them idempotent.

## Adding a migration

1. Create `backend/src/migrations/NNN_descriptive_name.sql` (3-digit prefix, sorted alphabetically).
2. Use `CREATE TABLE IF NOT EXISTS` / `INSERT OR IGNORE` so re-runs are safe.
3. Never edit an existing migration once it's been run anywhere — add a new file instead.
4. `npm run migrate` runs every `.sql` file in order. There is no down-migration framework yet (POC scope) — write forward-only.

## Reading / writing — patterns

```js
// SELECT
const user = db.get('SELECT * FROM users WHERE id = ?', [id]);
const users = db.all('SELECT * FROM users WHERE team_id = ?', [teamId]);

// INSERT
const r = db.run('INSERT INTO users (email, ...) VALUES (?, ...)', [email, ...]);
const newId = r.lastInsertRowid;

// UPDATE
db.run('UPDATE users SET first_name = ? WHERE id = ?', [name, id]);

// JSON columns
const stack = db.parseJson(row.tech_stack, []);     // safe read
const stackText = JSON.stringify(stack);            // write
```

## Don'ts

- Don't write raw SQL in route handlers if it spans more than 1 join + 1 condition — extract to a `models/` module (introduce when we need it, not yet).
- Don't add columns by editing the initial migration. Always add a new migration.
- Don't store secrets (tokens, keys) in any table without explicit hashing/encryption plan.
- Don't introduce an ORM (Sequelize/Prisma/Knex) without a sprint-level decision — current driver swap plan assumes raw SQL.
